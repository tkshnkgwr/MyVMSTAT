# リソース使用量記録 (FOOTPRINTS)

本ドキュメントは、**MyVMSTAT** の本番リリースビルドにおけるバイナリサイズ、メモリ使用量、および最適化設定についての計測値を記録します。

---

## 1. ビルド最適化設定 (Cargo.toml)

バイナリフットプリントとメモリオーバーヘッドを最小化するため、以下のリリース設定を適用しています。

```toml
[profile.release]
opt-level = 'z'      # サイズ優先の最適化
lto = true           # リンク時最適化 (Link-Time Optimization)
codegen-units = 1    # コンパイル単位を1つに統合して最適化を最大化
panic = 'abort'      # スタック巻き戻し(unwinding)を無効化しサイズ削減
strip = true         # シンボル情報とデバッグ情報を完全にバイナリから削除
```

---

## 2. 計測結果

### 2.1 環境スペック
- **OS**: Windows 11 (64-bit)
- **コンパイラ**: rustc 1.96.0 / cargo
- **依存クレート**: `sysinfo v0.39`, `chrono v0.4`, `windows v0.62`

### 2.2 メトリクス計測値

| 計測項目 | 計測値 | 説明 |
| :--- | :--- | :--- |
| **バイナリサイズ** | **206,336 バイト (約 201 KB)** | 依存関係を含むスタンドアロンな実行可能ファイルサイズ |
| **物理メモリ使用量 (Working Set)** | **約 15.9 MB** | 起動待機中のプロセスの物理メモリ（RSS相当）の占有量 |
| **仮想メモリサイズ** | **約 440 MB** | Windows 64-bit ランタイムの初期確保空間を含む仮想アドレスサイズ |
| **CPU 使用率** | **< 0.1%** | 周期的な1秒毎のポーリング処理時以外は、カーネル待機のためCPU消費なし |

---

## 3. 分析と評価

- **バイナリサイズ**: `windows` クレートや `sysinfo` クレートといった重量級のシステムAPIインターフェースをリンクしているにもかかわらず、`strip = true` および `opt-level = 'z'` により 201 KB という極めて小さいバイナリサイズを実現しました。
- **メモリ消費**: メモリリーク要因となる動的アロケーションを極力排除し、単一ループ内でバッファを再利用しているため、物理メモリ消費量は 15.9 MB で極めて安定しています。サーバー等での常時監視ツールとしても十分に実用に耐えうる省リソース性です。
